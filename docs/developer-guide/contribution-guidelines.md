# Contribution Guidelines

We welcome contributions to **SHKeeper** and appreciate your interest in improving the project. To keep the codebase stable, readable, and easy to maintain, please follow the guidelines below.

## Getting Started

1. Fork the repository on GitHub.
2. Create a new branch from `main` (or the default branch) for your changes.
3. Make sure your local environment is properly set up and the project builds successfully.

## Code Style & Quality

- Follow the existing code style and conventions used in the project.
- Keep changes focused and avoid unrelated refactoring in the same pull request.
- Write clear, readable, and well-documented code.
- Add comments where logic is complex or non-obvious.
- Ensure that your changes do not break existing functionality.

## Commits

- Use clear and descriptive commit messages.
- Keep commits small and logical.
- Avoid mixing multiple unrelated changes in a single commit.

## Pull Requests

- Open a pull request against the main repository.
- Clearly describe **what** was changed and **why**.
- Reference related issues if applicable.
- Make sure all checks and tests pass before requesting a review.

## Bug Reports & Feature Requests

- Use GitHub Issues to report bugs or suggest new features.
- Provide as much detail as possible:
  - Steps to reproduce the issue
  - Expected and actual behavior
  - Environment details (OS, version, deployment method)
- For feature requests, explain the use case and expected outcome.

## Documentation

- Improvements to documentation are always welcome.
- Ensure documentation changes are clear, concise, and accurate.
- Keep examples up to date with the current behavior of the system.

By contributing, you agree that your contributions will be licensed under the same license as the SHKeeper project.

Thank you for helping make SHKeeper better!
